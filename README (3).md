## 1. ปัญหาทางธุรกิจ

**ปัญหาของ Azure Stay — ต้นทุนการกระจายสูง (Channel Profitability)**

โรงแรม Azure Stay เผชิญกับกับดักคลาสสิกของอุตสาหกรรมโรงแรม: **ยอดการจองสูงแต่กำไรสุทธิลดลง** แม้ว่าโรงแรมจะขายห้องได้อย่างสม่ำเสมอผ่าน OTA (Online Travel Agency) หลายแห่ง เช่น Booking.com และ Expedia แต่รายได้จากการจองแต่ละครั้งจะถูกหักเป็น **ค่าคอมมิชชัน 15–20%** กลับไปให้แพลตฟอร์มบุคคลที่สาม

**สาเหตุหลัก:** โรงแรมเน้นเพิ่มประสิทธิภาพ *รายได้รวม* และ *อัตราการเข้าพัก* แต่ไม่ได้มุ่งเน้น *รายได้สุทธิ* ห้องที่จองผ่าน Booking.com ราคา ฿10,000 จะได้รับจริงเพียง ฿8,500 หลังหักค่าคอมมิชชัน 15% ในขณะที่ห้องห้องเดียวกันที่จองผ่านเว็บไซต์โดยตรงของโรงแรมจะได้รับเต็ม ฿10,000

**คำถามหลัก:** ช่องทางการจองใดที่ **เพิ่มกำไรสุทธิได้มากที่สุด** ไม่ใช่แค่ปริมาณการจอง?

## 2. วัตถุประสงค์แบบ SMART

| ตัวอักษร | วัตถุประสงค์ |
|----------|-------------|
| **S** (Specific — เฉพาะเจาะจง) | ระบุว่าช่องทางการจองใดเพิ่ม Net ADR สูงสุดและลดต้นทุนการหาลูกค้าต่ำที่สุด |
| **M** (Measurable — วัดได้) | เพิ่ม Net RevPAR **10%** และลดอัตราส่วนต้นทุนค่าคอมมิชชันรวม **5%** |
| **A** (Achievable — ทำได้จริง) | โดยการจัดสรรสินค้าคงคลังจาก OTA ที่มีค่าคอมมิชชันสูงไปยังช่องทางตรงและช่องทางต้นทุนต่ำ |
| **R** (Relevant — เกี่ยวข้อง) | สอดคล้องกับเป้าหมายของ Azure Stay ในการเพิ่มรายได้สุทธิและควบคุมต้นทุน |
| **T** (Time-bound — มีกำหนดเวลา) | วัดผลได้ภายใน **1 ไตรมาส (3 เดือน)** หลังดำเนินการตามกลยุทธ์ |

## 3. สมมติฐานและวิธีการ

### สมมติฐานที่ 1: ช่องทางตรงให้ Net ADR สูงกว่าช่องทาง OTA
**สูตรคำนวณ:**
```
Net ADR (ตามช่องทาง) = ผลรวม(net_room_revenue) / จำนวน(booking_id)
                       [กรองเฉพาะ status ≠ 'Cancelled']
```
**เกณฑ์ตัดสิน:** ถ้า Direct Net ADR > OTA Net ADR → ควรย้ายสินค้าคงคลังไปช่องทางตรง

---

### สมมติฐานที่ 2: โมเดลค่าคอมมิชชันแบบ Flat Fee มีประสิทธิภาพด้านต้นทุนสูงกว่าโมเดล Percentage
**สูตรคำนวณ:**
```
Cost % (ตามโมเดลค่าคอมมิชชัน) = [ผลรวม(commission_amount) / ผลรวม(gross_room_revenue)] × 100
```
**เกณฑ์ตัดสิน:** Cost % ต่ำกว่า = โรงแรมเก็บรายได้ได้มากกว่าต่อการจอง

---

### สมมติฐานที่ 3: ช่องทางที่มีอัตราการยกเลิกสูงสร้างต้นทุนเสียโอกาสอย่างมีนัยสำคัญ
**สูตรคำนวณ:**
```
อัตราการยกเลิก = COUNT(booking_id WHERE status = 'Cancelled') 
                  / COUNT(booking_id ทั้งหมด)   [จัดกลุ่มตาม channel_id]

รายได้ที่สูญเสีย = ผลรวม(gross_room_revenue WHERE status = 'Cancelled')
```
**เกณฑ์ตัดสิน:** ช่องทางที่มีอัตราการยกเลิก > 15% ต้องการการแก้ไขนโยบาย

## 4. Schema ข้อมูลและ Prompt ที่ใช้กับ AI

### 4.1 คำอธิบายวิธีใช้ AI สร้างชุดข้อมูล

ในการทำโปรเจกต์นี้ กลุ่มของเราใช้ **Claude AI (Anthropic)** ในการสร้างชุดข้อมูลสังเคราะห์ที่มีความสมจริง เนื่องจากข้อมูลการจองโรงแรมจริงมีข้อจำกัดด้านความเป็นส่วนตัว

**หลักการเขียน Prompt ที่ใช้:**

เราออกแบบ Prompt ตามแนวทาง **"Structure-First Prompting"** ซึ่งประกอบด้วย 4 องค์ประกอบหลัก ได้แก่:
1. **บริบท (Context):** บอก AI ว่าชุดข้อมูลนี้คืออะไร ใช้กับธุรกิจใด และมีวัตถุประสงค์อะไร
2. **Schema ที่ชัดเจน:** กำหนดชื่อคอลัมน์ ประเภทข้อมูล ช่วงค่า และความสัมพันธ์ระหว่างตาราง (Foreign Key) ให้ครบถ้วน
3. **สูตรคำนวณ:** ระบุวิธีคำนวณค่าที่ derive มาเพื่อให้ข้อมูลสอดคล้องกัน เช่น `net_room_revenue = gross - commission`
4. **สัดส่วนและความน่าจะเป็น:** กำหนดการกระจายของข้อมูล เช่น อัตราการยกเลิก สัดส่วนสถานะ ช่วง ADR

### 4.2 Prompt ที่ใช้สร้างชุดข้อมูล

```
สร้างชุดข้อมูลการจองโรงแรมที่สมจริงสำหรับโรงแรม 4 ดาวในกรุงเทพฯ 
ชื่อ "Azure Stay" สำหรับปี 2567 ข้อมูลต้องเป็นไปตาม schema นี้อย่างเคร่งครัด 
และต้องสอดคล้องกันภายใน (Foreign Key ต้องตรงกัน การคำนวณทางการเงินต้องถูกต้อง):

ตารางที่ 1: fact_bookings (1,150 แถว)
  - booking_id (PK): รูปแบบ "BK_00001"
  - booking_date: วันที่สุ่มในปี 2567 (YYYY-MM-DD H:MM)
  - check_in_date: 1-60 วันหลัง booking_date
  - channel_id (FK → dim_channels): CH_01 ถึง CH_06
  - Channel Name: ชื่อช่องทางตาม channel_id
  - rate_code_id (FK → dim_rate_codes): RT_RACK, RT_CORP, RT_PROMO, RT_NET
  - gross_room_revenue: ตาม ADR baseline ของ rate code (฿2,500–฿5,500/คืน) × จำนวนคืน
  - Commission Rate: อัตราคอมมิชชันตามโมเดลของช่องทาง
  - commission_amount: คำนวณตามโมเดล (0 สำหรับ Flat Fee, Fixed 500 สำหรับ GDS, % สำหรับ OTA)
  - net_room_revenue: คำนวณเป็น gross_room_revenue − commission_amount
  - status: "Checked-Out" (~63%), "Confirmed" (~23%), "Cancelled" (~14%)
            [ความน่าจะเป็น Cancelled สูงขึ้นสำหรับ RT_PROMO ~22-28%]

ตารางที่ 2: dim_channels (6 แถว)
  - CH_01: Booking.com,   OTA,       Percentage, 0.15, Alice
  - CH_02: Expedia,       OTA,       Percentage, 0.18, Bob
  - CH_03: Direct Web,    Direct,    Flat Fee,   0.00, Charlie
  - CH_04: GDS (Amadeus), Wholesale, Flat Fee,   500,  Diana
  - CH_05: Hotelbeds,     Wholesale, Merchant,   0.20, Eve
  - CH_06: Walk-in,       Direct,    Flat Fee,   0.00, Charlie

ตารางที่ 3: dim_rate_codes (4 แถว)
  - RT_RACK:  Rack Rate,    commissionable=TRUE
  - RT_CORP:  Corporate,    commissionable=TRUE
  - RT_PROMO: Promotional,  commissionable=TRUE  [ยกเลิกสูง!]
  - RT_NET:   Wholesale Net, commissionable=FALSE

ตารางที่ 4: fact_marketing_spend (48 แถว — 4 แพลตฟอร์ม × 12 เดือน)
  - spend_id, spend_date, channel_id=CH_03 (Direct Web เท่านั้น)
  - platform: Google Ads, Facebook, Instagram, Line Ads
  - cost_amount: ค่าใช้จ่ายรายเดือนต่อแพลตฟอร์ม (รวมประมาณ ฿224,000/ปี)
  - clicks: cost_amount / CPC สุ่มระหว่าง ฿8-15
```

### 4.3 คำอธิบาย Metric หลัก

| Metric | สูตร | หมายเหตุ |
|--------|------|----------|
| **Gross ADR** | `gross_room_revenue / จำนวนการจอง` | ก่อนหักค่าคอมมิชชัน |
| **Net ADR** | `net_room_revenue / จำนวนการจอง` | หลังหักค่าคอมมิชชัน — กำไรจริง |
| **Commission Cost %** | `commission_amount / gross_room_revenue × 100` | ต่ำกว่า = ดีกว่า |
| **อัตราการยกเลิก** | `การจองที่ยกเลิก / การจองทั้งหมด × 100` | ตามช่องทาง |
| **True COA %** | `(commission + marketing_spend) / gross_revenue × 100` | สำหรับช่องทางตรง |
| **Net RevPAR** | `(gross_revenue − commission) / จำนวนห้องทั้งหมด` | Metric ภาพรวม |

## 6. ข้อค้นพบสำคัญ

### 🔍 ข้อค้นพบ 1 — กับดักค่าคอมมิชชัน
ช่องทาง OTA (Booking.com, Expedia) คิดเป็น **~33% ของปริมาณการจองทั้งหมด** แต่ Net ADR ต่ำกว่าช่องทางตรง **15–18%** เนื่องจากค่าคอมมิชชัน ห้องที่จองผ่าน Booking.com Net ADR ประมาณ ฿8,300 เทียบกับ ฿12,300 สำหรับ Direct Web — ต่างกันเกือบ ฿4,000 ต่อการจอง

### 🔍 ข้อค้นพบ 2 — Flat Fee ได้เปรียบ
ช่องทาง Flat Fee (Direct Web, Walk-in) ต้นทุนค่าคอมมิชชันจริง **0%** เทียบกับโมเดล Percentage เฉลี่ย **~16-18%** แม้นับรวมค่าการตลาดประมาณ ฿224,000/ปีแล้ว True COA ของ Direct Web ยังต่ำกว่าค่าคอมมิชชัน OTA อย่างชัดเจน

### 🔍 ข้อค้นพบ 3 — ความเสี่ยงจากการยกเลิก
ช่องทาง OTA ที่เชื่อมกับ `RT_PROMO` (อัตราโปรโมชัน แบบยกเลิกฟรี) มีอัตราการยกเลิก **14–22%** นี่คือการสูญเสียซ้ำซ้อน: โรงแรมจ่ายค่าใช้จ่ายในการหาลูกค้า (ผ่านอันดับ OTA) แต่ยังสูญเสียสล็อตสินค้าคงคลังให้ผู้เข้าพักที่ไม่มาตามนัด

## 7. ข้อเสนอแนะ

### ✅ ข้อเสนอแนะที่ 1: เปลี่ยนนโยบายอัตราโปรโมชัน → Non-Refundable
**การดำเนินการ:** แปลง `RT_PROMO` (ยกเลิกฟรี) เป็น:
- Non-refundable ต้องชำระเงินล่วงหน้าเต็มจำนวน หรือ  
- กำหนดให้วางมัดจำ 30–50% ณ เวลาจอง

**ผลกระทบ:** ลดอัตราการยกเลิกจาก ~22% เหลือ ~4% ช่วยปรับปรุงความแม่นยำในการพยากรณ์รายได้และอัตราการเข้าพักจริง

---

### ✅ ข้อเสนอแนะที่ 2: โอนงบ OTA มาลงทุนช่องทางตรง
**การดำเนินการ:** นำ 10% ของงบค่าคอมมิชชัน OTA มาลงทุนใน:
- Google Ads สำหรับ Branded Keywords (เช่น "Azure Stay กรุงเทพ")
- ป้าย Best Rate Guarantee บนเว็บไซต์โรงแรม
- โปรแกรมความภักดี: ส่วนลด 5% + อาหารเช้าฟรีสำหรับลูกค้าที่จองตรง

**ผลกระทบ:** การย้าย 50 การจอง/ไตรมาสจาก Booking.com (฿8,300 สุทธิ) ไปยัง Direct Web (฿12,300 สุทธิ) = **รายได้สุทธิเพิ่มขึ้นกว่า ฿200,000 ต่อไตรมาส**

### ✅ ข้อเสนอแนะที่ 3: ขยายสัญญาองค์กรและ B2B
**การดำเนินการ:** เจ้าของสัญญา (Alice, Bob) ควรให้ความสำคัญกับการเซ็นสัญญาอัตรา Corporate (`RT_CORP`) กับบริษัทในกรุงเทพฯ เสนอราคาแบบ Tiered: ยิ่งปริมาณมาก ยิ่งได้อัตราดี
